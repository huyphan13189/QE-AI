# M2.3 — Viết & bảo trì automation có AI hỗ trợ

> **Level:** 2 · **Bắt buộc:** tùy chọn nâng cao (cần chút code) · **Skill liên quan:** [`writing-automation-tests`](../../.claude/skills/writing-automation-tests/SKILL.md) **(đã có)**

## 1. Mục tiêu năng lực

Sau module này, "Tôi có thể…"

- Dùng AI/coding assistant để sinh nhanh khung một automation test (ưu tiên API test), thay vì gõ tay toàn bộ boilerplate request/assertion.
- Kiểm chứng một test AI sinh ra là **test thật** — nghĩa là nó PASS khi hành vi đúng và **FAIL đúng** khi có bug — chứ không phải một test "luôn xanh" vì thiếu assertion hoặc assert sai điều kiện.
- Giải thích được khái niệm test tautological (test tự nó không thể fail, dù code có sai) và tự rà được test AI sinh có mắc lỗi này không.
- Viết và chạy được một API test cụ thể trên Toolshop để chứng minh một bug thật (giá âm không bị từ chối), dùng test đó như bằng chứng report bug.
- Bảo trì một test automation hiện có với AI hỗ trợ mà không làm mất coverage gốc — biết đọc diff AI đề xuất trước khi merge, không tin mù.

## 2. Vì sao & khi nào dùng AI ở công đoạn này

**Lợi ích thật:** AI rất giỏi sinh nhanh phần "cơ khí" của automation — khung file test, cú pháp gọi request đúng framework (Playwright request context, REST client, supertest…), boilerplate setup/teardown, gợi ý endpoint dựa trên Swagger/OpenAPI spec, và cả gợi ý selector khi test UI. Việc này tiết kiệm thời gian tra cú pháp, đặc biệt khi bạn quen một framework nhưng cần viết bằng framework khác, hoặc mới học automation.

**Cạm bẫy cụ thể, không phải lý thuyết suông:**

- **Test luôn xanh (always-green test).** AI sinh test gọi đúng API, nhận response, nhưng assertion lỏng lẻo kiểu `expect(response).toBeDefined()` hoặc `expect(response.status).not.toBe(500)` — test này pass ngay cả khi API trả về sai hoàn toàn so với kỳ vọng nghiệp vụ. Đây là lỗi phổ biến nhất và nguy hiểm nhất, vì test "xanh" tạo cảm giác an toàn giả.
- **Assertion sai hướng.** AI có thể hiểu nhầm yêu cầu và viết test khẳng định *hành vi hiện tại* (kể cả khi hành vi đó là bug) thay vì *hành vi mong muốn* — ví dụ nếu API đang sai (chấp nhận giá âm), AI "quan sát" response 200 rồi viết `expect(status).toBe(201)`, vô tình biến bug thành "spec".
- **Selector/endpoint giòn (brittle).** Với UI test, AI hay gợi ý selector theo text hiển thị hoặc cấu trúc DOM dễ đổi (`.nth(2)`, class CSS tạo tự động) thay vì `data-testid` ổn định — test vỡ ngay khi UI đổi nhẹ, dù hành vi nghiệp vụ không đổi.
- **Phụ thuộc quá mức, không đọc code AI sinh.** Copy-paste nguyên khối rồi chạy `npm test` thấy xanh là xong — bỏ qua bước quan trọng nhất: tự hỏi "test này có thật sự fail khi có bug không?".

**Quy tắc bắt buộc, không thương lượng:** mọi test do AI sinh ra, trước khi merge, phải được chứng minh **có thể fail** — bằng cách chạy nó trên một trạng thái sai (bug thật, hoặc tạm sửa code/mock để giả lập sai) và xác nhận test đỏ đúng chỗ. Một test không bao giờ thấy đỏ là một test vô giá trị, bất kể ai viết ra nó.

**Khi nào KHÔNG nên dùng AI ở bước này:** khi bạn chưa tự xác định rõ **hành vi mong đợi** (expected behavior) là gì — AI không thể đoán đúng assertion nếu chính bạn còn mơ hồ về oracle. Lúc đó cần quay lại phân tích requirement (M2.1) trước, rồi mới nhờ AI viết test.

## 3. Lý thuyết ngắn

Quy trình an toàn để viết automation có AI hỗ trợ gồm 4 bước, làm theo đúng thứ tự — bỏ qua bước nào cũng làm tăng rủi ro test vô giá trị:

1. **Mô tả rõ hành vi mong đợi trước khi nhờ AI viết code.** Viết ra bằng ngôn ngữ tự nhiên: input gì, output/status/body kỳ vọng là gì, điều kiện fail là gì. Đây chính là oracle — nếu bạn không viết được câu này, đừng nhờ AI viết test.
2. **Để AI sinh test dựa trên mô tả đó.** Đưa prompt cụ thể (endpoint, method, payload, framework muốn dùng). AI trả về code test.
3. **Đọc lại và sửa assertion — không copy-paste nguyên khối.** Kiểm tra: assertion có đúng khẳng định điều bạn mô tả ở bước 1 không, hay AI đã tự suy diễn thêm/bớt điều kiện? Đây là bước hay bị bỏ qua nhất.
4. **Chạy thử CẢ hai chiều: case pass và case fail.** Chạy với input hợp lệ → test phải xanh. Sau đó chạy với input mô phỏng đúng lỗi cần bắt (hoặc chính API đang có bug) → test phải đỏ. Nếu ở chiều thứ hai test vẫn xanh, assertion sai — quay lại bước 3.

**Khái niệm test API cơ bản** cần nắm để làm bước 2–4: một API test gửi một HTTP request (method + URL + headers + body) tới endpoint, nhận về response, rồi **assert** trên hai phần chính:

- **Status code** — ví dụ `4xx` (client error, do input sai) khác với `2xx` (thành công) khác với `5xx` (lỗi server ngoài ý muốn).
- **Response body** — ví dụ có field `errors` chứa message giải thích trường nào sai, hoặc field dữ liệu trả về đúng những gì gửi lên.

Một test API "kiểm chứng validate" tốt luôn assert **cả status lẫn (khi có thể) nội dung lỗi**, không chỉ mỗi status — vì API có thể trả đúng status nhưng sai message, hoặc ngược lại.

## 4. Hands-on trên sample project

Dùng kịch bản **S3** trong [BUGS.md](../sample-project/BUGS.md): API `POST /products` của Toolshop (`http://localhost:8091`, Swagger tại `http://localhost:8091/api/documentation`) hiện thiếu validate giá âm / trường bắt buộc. Ta sẽ viết một API test để **kiểm chứng bằng automation** rằng lỗ hổng này tồn tại — biến nó thành bằng chứng report bug, không chỉ là quan sát tay một lần.

### Bước 1 — Mô tả hành vi mong đợi (trước khi hỏi AI)

Viết ra rõ ràng oracle cần kiểm chứng:

> Khi gọi `POST /products` với `price` là số âm (ví dụ `-10`), server **phải** từ chối request với status `4xx` (ví dụ `422 Unprocessable Entity` hoặc `400 Bad Request`), và response body nên chứa thông tin lỗi liên quan tới field `price`. Server **không được** trả `200`/`201` và tạo sản phẩm với giá âm.

### Bước 2 — Prompt nhờ AI sinh test

Dùng prompt sau với coding assistant (Claude Code, Copilot, hay chatbot bất kỳ), điều chỉnh framework theo stack dự án của bạn (ví dụ ở đây dùng Playwright's request context cho API test):

```
Tôi cần viết một API test bằng Playwright Test (dùng request context, TypeScript)
cho endpoint POST http://localhost:8091/api/products của Toolshop
(practice-software-testing).

Hành vi mong đợi cần kiểm chứng:
- Gửi POST /products với body có "price": -10 (giá âm), các field bắt buộc khác
  điền hợp lệ (is_location_offer: false, is_rental: false, tên sản phẩm hợp lệ,
  category_id/brand_id hợp lệ nếu bắt buộc — tra Swagger tại
  http://localhost:8091/api/documentation nếu cần).
- Server PHẢI trả về status 4xx (400 hoặc 422), KHÔNG được trả 200/201.
- Nếu server trả lỗi validate, kỳ vọng response body chứa thông tin liên quan
  field "price".

Viết 1 test file hoàn chỉnh:
1. Một test case gửi price = -10, assert status là 4xx (dùng
   expect(response.status()).toBeLessThan(500) là KHÔNG đủ — phải assert đúng
   khoảng 4xx, ví dụ toBeGreaterThanOrEqual(400) và toBeLessThan(500)).
2. Assert thêm: nếu status là 2xx thì test phải FAIL rõ ràng kèm message giải
   thích đây là bug (giá âm bị chấp nhận).
3. Comment giải thích assertion nào kiểm tra điều gì.
Không cần viết case giá dương hợp lệ — tôi tự thêm sau.
```

### Bước 3 — Đọc & sửa assertion trước khi chạy

Trước khi chạy, tự kiểm tra code AI trả về đúng 3 điều:

- Assertion có so status với khoảng `[400, 500)` cụ thể, không phải chỉ `!== 500` hay `toBeDefined()` (đây chính là dạng "test luôn xanh" nêu ở mục 2).
- Test không "âm thầm pass" khi request lỗi vì lý do khác (ví dụ sai payload format khiến 4xx do lỗi JSON chứ không phải do validate giá) — nếu nghi ngờ, log thêm response body ra console khi assertion fail.
- Nếu AI tự thêm bước dọn dữ liệu (xoá product vừa tạo) mà request thực ra phải bị từ chối, kiểm tra logic đó có hợp lý không — sản phẩm không nên được tạo ra, nên bước dọn dẹp thường là thừa hoặc là dấu hiệu AI ngầm giả định request sẽ thành công.

Ví dụ một phiên bản test đã rà và sửa (rút gọn để minh hoạ):

```ts
import { test, expect } from '@playwright/test';

test('POST /products từ chối giá âm với 4xx', async ({ request }) => {
  const response = await request.post('http://localhost:8091/api/products', {
    data: {
      name: 'Test hammer QE-AI',
      price: -10,               // giá âm — hành vi cần kiểm chứng
      is_location_offer: false,
      is_rental: false,
    },
  });

  const status = response.status();
  const body = await response.json().catch(() => null);

  // Assertion chính: server phải từ chối bằng 4xx, không phải 2xx hay 5xx.
  // Nếu status là 2xx, đây chính là bug S3 — test phải báo đỏ rõ ràng.
  expect(
    status,
    `Kỳ vọng 4xx khi price âm, nhưng nhận status ${status}, body: ${JSON.stringify(body)}`
  ).toBeGreaterThanOrEqual(400);
  expect(status).toBeLessThan(500);
});
```

### Bước 4 — Chạy thử cả hai chiều

- **Chạy ngay bây giờ trên Toolshop hiện tại:** nếu API đang có bug S3 (chấp nhận giá âm), test này sẽ **FAIL** — và đó chính xác là kết quả đúng, vì nó vừa tự động hoá việc phát hiện bug. Chụp lại output fail này làm bằng chứng report bug (kèm status code thật server trả về).
- **Kiểm tra test có thể pass đúng cách:** tạm sửa `price: -10` thành `price: -1` rồi thử với một API/mock giả lập trả `422` để xác nhận assertion cũng pass đúng khi hành vi đúng — hoặc đơn giản hơn, nếu team đã fix bug này ở phiên bản sau, chạy lại test đó và xác nhận nó chuyển xanh. Mục đích: chứng minh test **có khả năng cả pass lẫn fail**, không phải kiểu test luôn đỏ do lỗi cú pháp/kết nối.

Nếu bạn chạy thử và thấy Toolshop bản bạn clone **đã** validate đúng giá âm (bug đã fix), coi bước này là bài tập viết **regression test** — vẫn giữ nguyên giá trị: test này giờ đóng vai trò canh giữ, đảm bảo bug S3 không quay lại trong tương lai.

## 5. Checklist năng lực

| Tiêu chí | Tự đánh giá | Reviewer |
|----------|:-----------:|:--------:|
| Test chạy được (không lỗi cú pháp/kết nối, ra kết quả pass/fail rõ ràng) | ☐ | ☐ |
| Test FAIL đúng khi bug tồn tại — đã tự chứng minh test không phải dạng "luôn xanh" (đã thử cả input đúng lẫn input sai) | ☐ | ☐ |
| Assertion kiểm tra đúng hành vi mong đợi đã mô tả ở Bước 1 (đúng khoảng status 4xx, không chỉ `!== 500`) | ☐ | ☐ |
| Đã tự đọc và sửa ít nhất 1 điểm trong code AI sinh trước khi chạy (không copy-paste nguyên khối không kiểm tra) | ☐ | ☐ |
| Có thể giải thích bằng lời vì sao test này không phải test tautological | ☐ | ☐ |

## 6. Capstone — áp vào dự án thật

Chọn một hành vi **cần kiểm chứng bằng automation** trong dự án bạn đang làm thật (ưu tiên một API endpoint thiếu validate, hoặc một bug đã report nhưng chưa có test canh giữ). Với AI hỗ trợ, viết một automation test thật theo đúng quy trình 4 bước ở mục 3, rồi nộp cho reviewer:

- Mô tả hành vi mong đợi bạn viết ra **trước khi** hỏi AI (Bước 1).
- Prompt bạn dùng để nhờ AI sinh test.
- Code test AI sinh ra **ban đầu**, và code **sau khi bạn sửa** — ghi rõ đã sửa gì và vì sao (assertion lỏng lẻo đã siết lại thế nào, selector/endpoint nào bạn đổi vì giòn, v.v.).
- Bằng chứng đã chạy test ở cả hai chiều: kết quả khi hành vi đúng (pass) và khi hành vi sai/có bug (fail) — log hoặc screenshot output.
- Test này đã được thêm vào suite thật của dự án (đường dẫn file, hoặc PR link nếu có).

Capstone đạt yêu cầu khi reviewer có thể nhìn vào diff giữa bản AI sinh và bản bạn sửa, thấy rõ ràng bạn đã tự tay kiểm chứng — không phải chỉ "AI viết, tôi chạy, thấy xanh, xong".
