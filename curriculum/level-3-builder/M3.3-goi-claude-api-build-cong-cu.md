# M3.3 — Gọi Claude API/SDK để build công cụ test

> **Level:** 3 · **Bắt buộc:** nhánh nâng cao · **Skill liên quan:** [`building-test-tools-with-api`](../../.claude/skills/building-test-tools-with-api/SKILL.md) **(đã có)**

## 1. Mục tiêu năng lực

Sau module này, "Tôi có thể…"

- Viết được một script nhỏ (không cần UI, chạy dòng lệnh) gọi Claude API/SDK để tự động hoá **một việc QE cụ thể** — ví dụ: đọc mô tả một API endpoint của Toolshop rồi in ra bộ checklist test, thay vì phải mở chatbot và gõ tay prompt mỗi lần.
- Đọc API key từ biến môi trường một cách đúng đắn, giải thích được vì sao **tuyệt đối không hard-code key trong source code** và không commit key vào git dưới bất kỳ hình thức nào.
- Chọn được model phù hợp cho script của mình bằng cách tra cứu tài liệu/skill hiện hành tại thời điểm viết code, thay vì copy một ID model cố định từ tài liệu cũ (ID và giá model thay đổi theo thời gian).
- Đọc được response trả về từ API (lấy nội dung text, đọc số token đã dùng) và ước lượng sơ bộ chi phí của một lần gọi, thay vì chạy script mà không biết nó tốn bao nhiêu.
- Xử lý được lỗi cơ bản khi gọi API (mất kết nối, key sai, vượt rate limit) để script không crash im lặng hoặc treo vô thời hạn.

## 2. Vì sao & khi nào dùng AI ở công đoạn này

Đến module này, bạn đã quen dùng AI qua chatbot hoặc coding assistant — gõ prompt, đọc kết quả, làm tiếp bằng tay. Đó là cách đúng cho việc **làm một lần**. Nhưng có một lớp việc QE khác: việc bạn phải làm **lặp đi lặp lại theo cùng một khuôn** — mỗi khi có API endpoint mới thì cần một bộ checklist test tương ứng; mỗi khi có file requirement mới thì cần một bộ test case nháp ban đầu. Nếu mỗi lần đều mở chatbot, dán lại prompt, chỉnh sửa thủ công phần input, bạn đang trả giá bằng thời gian và sự nhất quán cho một việc đáng ra máy có thể làm hộ.

**Nên tự viết script gọi API khi:**
- Việc đó **lặp lại theo một khuôn cố định** — cùng một loại input (mô tả endpoint, file requirement), cùng một loại output mong muốn (checklist, bộ test case nháp) — và bạn dự đoán sẽ cần làm việc này nhiều lần trong nhiều tuần/tháng tới, không phải một lần duy nhất.
- Cần **tích hợp vào pipeline** — ví dụ script tự chạy mỗi khi có PR thay đổi file API spec, tự sinh checklist test đính kèm vào PR để reviewer đối chiếu, không cần ai phải nhớ mở chatbot.
- Cần **xử lý hàng loạt** — ví dụ có 20 endpoint trong Swagger spec, bạn muốn một lệnh chạy ra 20 bộ checklist thay vì copy-paste 20 lần vào chatbot.
- Cần **output có định dạng nhất quán** để máy đọc tiếp (ví dụ ghi thẳng ra file Markdown theo đúng khuôn mỗi lần, hoặc parse thành JSON để đẩy vào hệ thống quản lý test case).

**Chi phí và rủi ro cần cân nhắc trước khi viết script — đây không phải lựa chọn miễn phí:**
- **Chi phí token thật, tính tiền theo lượng gọi.** Chatbot trả phí cố định/tháng cho bạn dùng thoải mái; gọi API trực tiếp tính tiền theo số token input + output của **từng lần gọi**. Một script chạy tự động, lặp lại nhiều lần (đặc biệt nếu chạy trong CI mỗi commit) có thể tốn tiền nhanh hơn bạn tưởng nếu không kiểm soát — đây là lý do mục 3 có phần ước lượng chi phí.
- **Chi phí bảo trì.** Một prompt gõ tay trong chatbot có thể chỉnh ngay khi kết quả chưa ưng. Một script đã viết cần bạn quay lại sửa code khi output không đúng ý, khi API đổi (ví dụ tham số đổi tên), hoặc khi cần model mới — không "rẻ" như việc gõ lại một câu hỏi.
- **Rủi ro lộ API key.** Key gọi API tương đương một tấm thẻ tín dụng — ai có key đều gọi được và bạn trả tiền. Hard-code key trong file rồi lỡ commit lên git public là rò rỉ thật, đã xảy ra nhiều lần trong thực tế và bị lợi dụng gần như ngay lập tức. Mục 3 sẽ nói rõ cách tránh.
- **Rủi ro lộ dữ liệu nhạy cảm.** Nội dung bạn gửi trong prompt (requirement, log lỗi, dữ liệu test) rời khỏi máy bạn và đi tới nhà cung cấp API. Nếu requirement chứa thông tin khách hàng thật, dữ liệu production, hay bí mật kinh doanh chưa công bố, cần cân nhắc kỹ trước khi đưa nguyên văn vào script — quy tắc này áp dụng y hệt dù bạn gọi qua chatbot hay qua script, nhưng script tự động dễ khiến người ta quên rà lại nội dung mỗi lần chạy.

**Khi nào chatbot vẫn là lựa chọn tốt hơn:** việc một lần, cần bạn tương tác qua lại nhiều vòng để làm rõ ý (như mục 3 ở M2.1 — hỏi câu hỏi làm rõ trước khi sinh test case), hoặc bạn chưa chắc chắn hình dạng output mong muốn nên chưa đáng để "đóng cứng" thành code.

## 3. Lý thuyết ngắn

Một script gọi Claude API để làm việc QE gồm 5 phần bắt buộc phải hiểu đúng, theo đúng thứ tự khi viết code:

**1. API key — đọc từ biến môi trường, không có ngoại lệ.** Key không bao giờ được gõ thẳng thành chuỗi trong source code (`api_key = "sk-ant-..."` là sai tuyệt đối, kể cả khi bạn định xoá trước khi commit — dễ quên). Cách đúng: đọc từ biến môi trường hệ thống (`ANTHROPIC_API_KEY`), để terminal hoặc file cấu hình CI đặt biến này, script chỉ đọc ra bằng hàm đọc biến môi trường của ngôn ngữ đang dùng. Nếu cần lưu key khi phát triển local, dùng file `.env` **và thêm `.env` vào `.gitignore`** trước khi tạo file đó — không phải sau. Không bao giờ dán key thật vào tin nhắn chat, log, hay ảnh chụp màn hình khi hỏi đồng nghiệp/AI giúp debug.

**2. SDK chính thức, không tự gọi HTTP thô nếu ngôn ngữ có SDK.** Dùng thư viện chính thức của Anthropic cho ngôn ngữ bạn viết (ví dụ gói `anthropic` cho Python, `@anthropic-ai/sdk` cho JavaScript/TypeScript) thay vì tự dựng request bằng `requests`/`fetch`. SDK xử lý sẵn: định dạng request đúng chuẩn, phân loại lỗi (hết hạn mức, sai key, lỗi server), và retry tự động cho một số lỗi tạm thời — tự viết lại các phần này bằng tay dễ sai và không cần thiết.

**3. Chọn model — không hard-code ID cụ thể vào code lâu dài mà không kiểm tra lại.** ID model và bảng giá của Anthropic thay đổi theo thời gian (model mới ra mắt, model cũ ngừng hỗ trợ). Vì tài liệu này không "sống" theo thời gian thực, **không in ID model cụ thể ở đây** — khi viết code thật, tra cứu tài liệu Claude API hiện hành (hoặc skill `claude-api` nếu môi trường bạn có) để lấy đúng ID model đang được khuyến nghị tại thời điểm đó. Cách làm an toàn trong code: đặt ID model vào một biến cấu hình/biến môi trường riêng (ví dụ `CLAUDE_MODEL`), không viết chết trong logic gọi API — khi cần đổi model, chỉ đổi một chỗ.

**4. Gửi prompt và đọc response.** Request gửi lên gồm (tối thiểu): model đã chọn, giới hạn token output tối đa, và danh sách tin nhắn (`messages`) — trong đó có nội dung bạn muốn hỏi (ví dụ: mô tả một API endpoint kèm yêu cầu sinh checklist test). Có thể kèm thêm một "system prompt" — chỉ dẫn vai trò/quy tắc chung áp dụng cho toàn bộ yêu cầu (ví dụ: "Bạn là QA engineer, luôn trả lời dạng danh sách có đánh số"). Response trả về là danh sách các "khối nội dung" (content blocks) — script phải kiểm tra loại khối trước khi đọc nội dung text bên trong, vì response có thể chứa nhiều loại khối khác nhau tuỳ tính năng bật/tắt.

**5. Ước lượng chi phí bằng usage token.** Mỗi response trả kèm một mục thống kê số token đã dùng cho input (prompt bạn gửi) và output (nội dung model trả về). Vì đơn giá theo token thay đổi theo model và theo thời gian, không hard-code con số tiền cụ thể vào code — thay vào đó, in ra số token thực tế đã dùng ở mỗi lần gọi, và tra bảng giá hiện hành (tài liệu Claude API / skill `claude-api`) khi cần tính tiền chính xác. Với script chạy lặp lại nhiều lần (CI, batch), luôn log số token mỗi lần gọi để phát hiện sớm nếu chi phí tăng bất thường (ví dụ do gửi nhầm nguyên văn một file quá dài).

**Xử lý lỗi cơ bản:** script gọi API qua mạng nên luôn có khả năng gặp lỗi — key sai, hết hạn mức (rate limit), mất kết nối, timeout. Ở mức tối thiểu, bọc lệnh gọi API trong khối bắt lỗi (`try/except` hoặc tương đương) để: (a) in ra thông báo lỗi rõ ràng thay vì để chương trình crash với traceback khó hiểu, và (b) không để script "treo" vô thời hạn nếu không có giới hạn thời gian chờ (timeout). Không cần xử lý sâu từng loại lỗi ở bài tập này — chỉ cần đảm bảo lỗi được bắt và báo rõ, không bị nuốt im lặng.

## 4. Hands-on trên sample project

Bài tập này dùng kịch bản **S3** trong [BUGS.md](../sample-project/BUGS.md) — API `POST /products` của Toolshop (`http://localhost:8091`, Swagger tại `http://localhost:8091/api/documentation`). Thay vì tự tay đọc Swagger rồi gõ prompt vào chatbot như ở M2.3, ta viết **một script chạy được nhiều lần** — đưa mô tả endpoint vào, script tự gọi Claude API và in ra một bộ checklist test cho endpoint đó.

Trước khi bắt đầu: mở Swagger của Toolshop (`http://localhost:8091/api/documentation`), tìm mục mô tả `POST /products`, ghi lại các field trong request body (tên, kiểu dữ liệu, field nào bắt buộc) — đây chính là phần nội dung script sẽ gửi cho model.

### Bước 1 — Chuẩn bị: cài SDK và đặt key

```bash
pip install anthropic          # SDK chính thức cho Python
export ANTHROPIC_API_KEY="dán-key-thật-của-bạn-ở-đây"   # KHÔNG lưu key này vào bất kỳ file nào commit lên git
```

Nếu dùng file `.env` để tiện khi phát triển local, thêm `.env` vào `.gitignore` **trước** khi tạo file, và không bao giờ commit file đó dù chỉ một lần "để test thử".

### Bước 2 — Viết script gọi API

Script dưới đây đọc key và tên model từ biến môi trường (không hard-code cả hai), nhận mô tả endpoint qua một biến cố định trong script để bài tập gọn — trong dự án thật bạn có thể đổi thành đọc từ file/argument dòng lệnh:

```python
import os
import sys
from anthropic import Anthropic, APIError, APIConnectionError

# 1. Key: đọc từ biến môi trường — KHÔNG BAO GIỜ gõ chuỗi key thẳng vào đây.
api_key = os.environ.get("ANTHROPIC_API_KEY")
if not api_key:
    print("Lỗi: chưa đặt biến môi trường ANTHROPIC_API_KEY.")
    sys.exit(1)

client = Anthropic(api_key=api_key)

# 2. Model: đọc từ biến môi trường, có giá trị mặc định là placeholder.
# Khi viết code thật: tra cứu tài liệu Claude API hiện hành / skill "claude-api"
# để lấy đúng ID model đang được khuyến nghị tại thời điểm viết — ID model và
# giá thay đổi theo thời gian nên KHÔNG hard-code một ID cố định ở đây.
MODEL = os.environ.get("CLAUDE_MODEL", "<TRA-CUU-ID-MODEL-HIEN-HANH>")

SYSTEM_PROMPT = (
    "Bạn là một QA engineer có kinh nghiệm về API testing. "
    "Khi được đưa mô tả một endpoint, hãy trả lời bằng một checklist test "
    "dạng danh sách có đánh số, phân rõ 3 nhóm: Positive, Negative, Boundary. "
    "Không bịa ra field hay rule không có trong mô tả được cung cấp; nếu thiếu "
    "thông tin để viết một case cụ thể, ghi rõ '[CẦN XÁC NHẬN]' thay vì đoán."
)

# Mô tả endpoint lấy từ Swagger Toolshop — dán nguyên văn field/kiểu dữ liệu
# bạn đọc được, không tự thêm giả định vào phần input này.
endpoint_description = """
Endpoint: POST /products
Body (JSON):
- name (string, bắt buộc)
- price (number, bắt buộc)
- is_location_offer (boolean, bắt buộc)
- is_rental (boolean, bắt buộc)
- category_id (integer, tuỳ chọn)
- brand_id (integer, tuỳ chọn)
Không có mô tả validate cho "price" (chưa rõ có chặn giá âm hay không).
"""

try:
    response = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        system=SYSTEM_PROMPT,
        messages=[
            {"role": "user", "content": f"Sinh checklist test cho endpoint sau:\n{endpoint_description}"}
        ],
    )
except APIConnectionError:
    print("Lỗi: không kết nối được tới Claude API. Kiểm tra mạng và thử lại.")
    sys.exit(1)
except APIError as e:
    print(f"Lỗi khi gọi Claude API: {e}")
    sys.exit(1)

# 3. Đọc response: kiểm tra loại khối trước khi đọc .text
for block in response.content:
    if block.type == "text":
        print(block.text)

# 4. Ước lượng chi phí sơ bộ: chỉ in số token thực tế dùng — tra bảng giá
# hiện hành khi cần quy đổi ra tiền, không hard-code đơn giá vào code.
usage = response.usage
print(f"\n--- Token đã dùng: input={usage.input_tokens}, output={usage.output_tokens} ---")
```

### Bước 3 — Đối chiếu kết quả với Swagger thật

Đọc checklist script vừa in ra, đối chiếu từng dòng với đúng nội dung Swagger đã ghi lại ở đầu bài tập:

- Case nào nhắc tới field không có trong mô tả bạn đã dán vào `endpoint_description` (ví dụ model tự bịa thêm field `sku` hay `warranty`) → đánh dấu là case bịa, cần loại.
- Case nào liên quan tới việc `price` âm — đây chính là điểm yếu S3 thật của Toolshop — kiểm tra checklist có nêu đúng case này ở nhóm Negative/Boundary không.
- Case nào được đánh dấu `[CẦN XÁC NHẬN]` — đây là dấu hiệu tốt (model không bịa khi thiếu thông tin), không phải lỗi.

## 6. Capstone — áp vào dự án thật

Chọn **một việc QE lặp lại thật** trong dự án bạn đang làm — việc mà bạn (hoặc team) phải làm đi làm lại theo cùng một khuôn, ví dụ: mỗi khi có endpoint API mới cần một bộ checklist test, mỗi khi có bug report mới cần tóm tắt nhanh mức độ nghiêm trọng, hoặc mỗi khi có file requirement cần bung ra bộ test case nháp ban đầu.

Viết một script nhỏ gọi Claude API/SDK để tự động hoá việc đó, theo đúng các yêu cầu ở mục 3 và 5 (key qua biến môi trường, model không hard-code, xử lý lỗi cơ bản, in usage token). Nộp cho reviewer:

- Mô tả ngắn việc QE lặp lại bạn chọn, và vì sao nó xứng đáng viết thành script thay vì tiếp tục dùng chatbot (đối chiếu với tiêu chí "khi nào nên viết script" ở mục 2).
- Source code script (đã xoá/ẩn key thật trước khi nộp — chỉ nộp code, không nộp key).
- Bằng chứng đã chạy thành công: output thật + số token đã dùng.
- Nếu script này sẽ được chạy lặp lại nhiều lần (CI, batch), ước tính sơ bộ chi phí cho một tuần/tháng sử dụng dự kiến, dựa trên số token quan sát được và bảng giá bạn tra cứu tại thời điểm nộp bài.

Capstone đạt khi reviewer xác nhận: script chạy được, key/model được cấu hình đúng cách (không hard-code, không lộ), và bạn giải thích được rõ ràng lý do việc này xứng đáng tự động hoá bằng script thay vì tiếp tục làm tay qua chatbot.
