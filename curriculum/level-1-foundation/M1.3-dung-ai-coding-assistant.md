# M1.3 — Dùng AI coding assistant an toàn

> **Level:** 1 · **Bắt buộc:** có · **Skill liên quan:** — (không có)

## 1. Mục tiêu năng lực

Sau module này, "Tôi có thể…"

- Dùng được một AI coding assistant (Claude Code / Cursor / GitHub Copilot…) ở mức cơ bản: hỏi về codebase, nhờ sinh/sửa file, nhờ chạy lệnh.
- Phân biệt được coding assistant tích hợp repo với chatbot thuần text — biết assistant có thể *đọc, sửa, chạy* chứ không chỉ trả lời.
- Luôn xem diff trước khi chấp nhận thay đổi, không để assistant tự commit khi chưa review.
- Cẩn trọng khi cấp quyền chạy lệnh: biết lệnh nào an toàn để auto-approve, lệnh nào phải tự tay xác nhận.

## 2. Vì sao & khi nào dùng AI ở công đoạn này

**Lợi ích so với chatbot thường:**

- Assistant đọc được *cả repo* — nó thấy cấu trúc thư mục, import, config thật, không chỉ đoạn code mình dán vào.
- Sửa file trực tiếp thay vì mình phải copy-paste tay từng đoạn trả lời.
- Chạy được lệnh: build, chạy test, chạy linter — nó có thể tự kiểm tra xem thay đổi có "chạy được" không, thay vì chỉ đoán.

**Rủi ro cần biết:**

- **Sửa nhầm chỗ mình không ngờ** — assistant có thể sửa file khác ngoài ý định (side effect), nhất là khi yêu cầu mơ hồ.
- **Chạy lệnh phá hoại** — `rm -rf`, `git reset --hard`, `git push --force`, migrate database… nếu cấp quyền auto-run vô tội vạ, một câu lệnh sai có thể xoá dữ liệu hoặc lịch sử git.
- **Lộ code/dữ liệu nội bộ** — nếu dùng bản cloud của assistant với repo có chứa secret, API key, dữ liệu khách hàng thật, cần biết chính sách dữ liệu của công cụ trước khi dán vào.
- **Ảo giác (hallucination) áp dụng vào code thật** — assistant có thể "tưởng tượng" ra một API, một field không tồn tại, rồi sửa code dựa trên tưởng tượng đó. Nếu không review, bug này lọt thẳng vào codebase.

**Khi nào NÊN để nó tự chạy (ít giám sát hơn):**

- Các lệnh đọc, không phá hoại: `git status`, `git diff`, chạy test ở môi trường local/sandbox, `ls`, `grep`.
- Việc lặp lại, rủi ro thấp: sinh file test mới (chưa tồn tại), format code, viết docstring.

**Khi nào KHÔNG nên để nó tự chạy:**

- Bất kỳ lệnh nào ghi/xoá dữ liệu, đổi lịch sử git (`push --force`, `reset --hard`), động vào production, hoặc commit/push code lên remote.
- Khi yêu cầu ảnh hưởng nhiều file cùng lúc — luôn xem diff trước.
- Khi làm việc trên nhánh chính (main/master) chưa có backup.

**Ví dụ thực tế để dễ hình dung**

- *Lợi ích*: bạn hỏi "trong Toolshop, endpoint nào xử lý đăng nhập?" — chatbot thường sẽ đoán dựa trên kiến thức chung về Laravel, còn coding assistant tích hợp repo sẽ tìm đúng file route, đúng controller, trả lời chính xác theo code thật.
- *Rủi ro*: bạn yêu cầu "dọn dẹp code thừa trong thư mục tests" nhưng không nói rõ phạm vi — assistant có thể xoá nhầm một test đang dùng thật vì tưởng nó "thừa". Nếu không xem diff, bạn sẽ mất test đó mà không hay biết cho đến khi CI báo lỗi hoặc tệ hơn là không ai phát hiện.
- *Rủi ro dữ liệu*: nếu repo có file `.env` chứa API key hoặc thông tin khách hàng thật, và bạn dùng bản cloud của assistant, cần kiểm tra chính sách dữ liệu trước — nhiều đội chọn cách thêm các file nhạy cảm vào ignore-list của assistant hoặc dùng bản chạy local.

## 3. Lý thuyết ngắn

**Chatbot vs coding assistant tích hợp repo**

| | Chatbot (ChatGPT/Claude web) | Coding assistant (Claude Code, Cursor, Copilot Chat…) |
|---|---|---|
| Input | Mình copy-paste code vào | Đọc trực tiếp file trong repo |
| Output | Chỉ trả về text | Có thể sửa file, tạo file, chạy lệnh |
| Ngữ cảnh | Giới hạn ở những gì mình dán | Thấy cả cấu trúc dự án, import, config |
| Rủi ro | Thấp — chỉ đọc câu trả lời | Cao hơn — có "tay" để hành động |

**Khái niệm quyền hạn & chế độ permission**

Coding assistant thường có các mức quyền, ví dụ:

- **Read-only**: chỉ được đọc file, không sửa, không chạy lệnh ghi.
- **Ask before edit/run**: mỗi lần sửa file hoặc chạy lệnh đều phải mình bấm "chấp nhận".
- **Auto-approve (theo danh sách hoặc toàn phần)**: assistant tự chạy mà không hỏi — nhanh nhưng rủi ro nếu cấu hình sai.

Nguyên tắc: quyền càng rộng, review càng phải kỹ. Người mới nên bắt đầu ở chế độ "ask before edit/run" cho đến khi hiểu rõ công cụ.

**Luồng làm việc an toàn**

```
Yêu cầu (prompt rõ ràng)
      ↓
Assistant đề xuất thay đổi (diff / lệnh sẽ chạy)
      ↓
MÌNH XEM DIFF / XEM LỆNH  ← bước không được bỏ qua
      ↓
Review: đúng ý không? có sửa nhầm chỗ khác không? có lệnh nguy hiểm không?
      ↓
Giữ lại (apply) hoặc yêu cầu sửa lại — RỒI mới commit
```

Ba việc không bao giờ được bỏ qua: (1) đọc diff trước khi apply, (2) không commit khi chưa xem, (3) không auto-approve lệnh mình không hiểu.

**Một mẹo nhỏ nhưng quan trọng**: prompt càng cụ thể (nói rõ phạm vi file, hành vi mong muốn, ràng buộc "không sửa gì ngoài X"), assistant càng ít có cơ hội "tự do diễn giải" sai ý. Yêu cầu mơ hồ là nguyên nhân phổ biến nhất khiến assistant sửa nhầm chỗ không ai ngờ.

## 4. Hands-on trên sample project

Bài tập chạy trên [Toolshop](../sample-project/README.md).

**Bước 0 — chuẩn bị**

```bash
git clone https://github.com/testsmith-io/practice-software-testing.git
cd practice-software-testing
docker compose up -d
```

Mở thư mục này bằng một AI coding assistant (Claude Code, Cursor, hoặc Copilot Chat trong VS Code).

**Bước 1 — hỏi hiểu code lạ**

Prompt mẫu:

```
Giải thích luồng checkout trong code này: từ lúc người dùng bấm "Đặt hàng"
ở frontend cho tới khi order được lưu ở backend. Liệt kê các file/endpoint
chính liên quan, không sửa gì cả, chỉ đọc và giải thích.
```

Đọc câu trả lời, đối chiếu lại với 1-2 file nó nhắc tới để tự xác nhận nó nói đúng (không tin mù).

**Bước 2 — yêu cầu thêm một test đơn giản**

Prompt mẫu:

```
Thêm một test đơn giản kiểm tra: khi thêm sản phẩm vào giỏ hàng với
số lượng âm (quantity = -1), hệ thống phải từ chối hoặc không cho thêm.
Chỉ tạo file test mới, không sửa code hiện có. Cho tôi xem diff trước khi
tôi quyết định giữ hay không.
```

**Bước 3 — review diff (bắt buộc, không được bỏ qua)**

Trước khi bấm "giữ/apply", tự hỏi:

- File nào bị thay đổi? Có đúng như mình yêu cầu (chỉ file test mới) không?
- Assertion trong test có hợp lý không, hay assistant "đoán" hành vi mà không kiểm tra thực tế?
- Có dòng lệnh nào nó định chạy mà mình không hiểu không (ví dụ lệnh xoá, lệnh migrate)?

Chỉ khi trả lời ổn cả ba câu trên mới giữ lại thay đổi. Không commit ngay — để đó, tự chạy test bằng tay một lần nữa nếu có thể.

| Giải thích được khác biệt giữa chatbot thường và coding assistant tích hợp repo | ☐ | ☐ |

## 6. Capstone — áp vào dự án thật

Dùng AI coding assistant làm một việc nhỏ, có thật trên dự án bạn đang làm (ví dụ: thêm một test case, viết một script kiểm tra dữ liệu, sửa một đoạn code style nhỏ).

Nộp cho reviewer:

- Prompt bạn đã dùng.
- Diff/thay đổi assistant đề xuất (trước khi bạn chỉnh sửa gì).
- Ghi lại ngắn gọn: bạn đã review những gì trước khi giữ lại (đúng phạm vi file? assertion/logic có hợp lý? có lệnh nguy hiểm nào bị chặn lại không?).
- Kết quả cuối: đã giữ nguyên, chỉnh sửa thêm, hay bỏ đề xuất của assistant — và vì sao.
